Intent-Driven Rental Search

Prototype Case Study · AI Agents · Product Design

Renting is one of those problems people complain about constantly but rarely stop to redesign properly. Most rental platforms still assume that if users are given enough filters, they’ll eventually find what they want.

In practice, that rarely happens. People don’t think in parameters; they think in intent.

While applying for a high-agency role at an early-stage proptech startup, I decided to build a small prototype to explore whether a different approach could reduce friction in the rental journey. Instead of starting with features or metrics, I started with a simple question: Why is it still so hard to go from “I want a place” to “I’ve booked a viewing”?

The Translation Problem

From my own experience and from watching how people actually search for homes, the core issue seemed to be translation. Users know what they want, but they struggle to express it in the rigid language platforms expect.

Someone might say they want a quiet apartment near public transport, within a certain budget, and available soon. What the platform hears is a set of incomplete filters that either return nothing or flood the user with irrelevant results.

Hypothesis: Intent over Precision

The hypothesis behind the prototype was straightforward: if users can express intent in natural language, the system should adapt to them rather than the other way around. Instead of forcing precision early, it should accept ambiguity, make trade-offs explicit, and help users move forward quickly.

The prototype uses a chat-style interface where users describe what they’re looking for in plain language. That input is parsed into a set of structured signals—budget constraints, location preferences, timing, and softer lifestyle requirements.

Importantly, these signals aren’t treated as absolute filters. They’re weighted and used to score listings relative to one another. This ranking-first approach was a deliberate choice.

In real rental decisions, people rarely find a perfect match. They make compromises. By ranking listings and explaining why something shows up higher or lower, the system mirrors how users actually evaluate options.

Reality Checks & Closing the Loop

Another issue I wanted to address was unrealistic expectations. Many rental searches fail not because there are no options, but because the user’s budget doesn't align with the market.

To handle this, I added a Market Check Layer. When requirements aren't realistic, the system explains what’s off and suggests what to relax. When they are, it moves the user forward immediately.

Speed is critical in rental markets, so the prototype doesn’t stop at showing results. Once a user finds a suitable option, the system drafts a viewing request email to the landlord in real time, removing that small but meaningful delay between intent and action.

Comparing Approaches

Traditional Filter Search Intent-Driven AI Search
Requires users to translate needs into "platform language." Allows users to express intent using natural language.
Forces precision early in the journey. Accepts ambiguity; helps make trade-offs later.
Binary "match or no match" outcomes. Ranked outcomes based on weighted signals.
Process ends at displaying results. Bridges gap to action by drafting viewing emails.

Why I built this

The prototype was built using mock data and simple heuristics rather than complex models. The goal wasn’t accuracy at scale, but clarity of thought. I wanted to make the decision logic inspectable and easy to iterate on, rather than hiding everything behind a black box.

"Most friction in marketplaces doesn’t come from lack of features, but from poorly designed decision-making."

This project reflects how I prefer to work. Instead of writing speculative docs, I tend to build small, concrete systems to test assumptions. It was a way to think clearly about a broken process and explore how intent-driven systems could meaningfully reduce time-to-rent.
